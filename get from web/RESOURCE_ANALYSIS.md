## 逐路径资源评估

| 路径 | 内存需求 | 磁盘需求 | CPU | 风险 |
|------|---------|---------|-----|------|
| **Path 1** Playwright | Chrome 500MB-1GB | Chrome 数据 ~300MB + 截图 ~100MB | 中 | 🟡 |
| **Path 2** HTTP API | ~100MB | <50MB | 低 | 🟢 |
| **Path 3** 菁优网 Playwright | Chrome 500MB-1GB | Chrome 数据 ~300MB | 中 | 🟡 |
| **Path 4** 公开数据集 | ~200MB | 可能 1-5 GB（数据集下载） | 低 | 🔴 磁盘 |

| **Path 6** XKW Playwright | Chrome 500MB-1GB | Chrome 数据 ~300MB | 中 | 🟡 |
| **Path 7** App API 逆向 | ~300MB（mitmproxy 等） | ~200MB（工具安装） | 低 | 🟢 |
| **Path 8** 文档搜索 | ~200MB | 可能 500MB-2GB（PDF/DOC） | 低 | 🟡 磁盘 |

### 致命场景

**如果 Path 1/3/6 同时跑 Playwright/Chrome**：3 个 Chrome 实例 = 1.5-3 GB RAM，加上系统已用的 6 GB，会严重超出 8 GB 物理内存，导致激烈换页，系统接近不可用。

**如果 Path 4 下载大型数据集 + Path 8 下载大量文档**：13 GB 剩余空间可能被打到 5 GB 以下，影响系统正常运行。

## 调整方案

### Path 1、3、6（所有 Playwright 路径）：串行化，共享浏览器

三条路径都需要 Playwright + Chrome，不能同时跑。调整为：

- Path 1 先跑通（已有现成代码，最快）
- Path 3、6 复用 Path 1 的浏览器启动/登录机制，不要各自装一套 Chrome
- 提示词中加入：启动前先检查是否有其他路径正在用 Chrome，有则等待

### Path 4（数据集）：加磁盘配额

- 数据集下载前先评估大小，超过 2 GB 的数据集不下载，改用采样
- 优先选 HuggingFace 上可直接流式加载的数据集（不占本地磁盘）

### Path 8（文档搜索）：控制下载量

- 最多下载 5 份试卷作为验证，不要批量下载
- 验证完流程后立即清理中间文件

### 启动顺序建议

```
第一批（零资源冲突，可同时启动）：
  Path 2（HTTP API 逆向）
  Path 4（数据集搜索，先做调研不下载）
  Path 5（搜索+LLM）
  Path 7（App API 逆向调研）
  Path 8（文档平台调研）

第二批（等 Path 1 完成浏览器登录后）：
  Path 1（Playwright，先跑通登录+抓取）
  
第三批（复用 Path 1 的浏览器机制后）：
  Path 3（菁优网）
  Path 6（学科网试卷）
```
